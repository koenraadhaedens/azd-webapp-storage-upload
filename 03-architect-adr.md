## ADR-001: OTP Email Delivery via Azure Logic App + Office 365 Connector

**Status**: Accepted

**Context**: The portal needs to send a 6-digit OTP to users via email. Options considered:
1. Direct SMTP from App Service (requires SMTP credentials)
2. Azure Communication Services Email
3. Azure Logic App + Office 365 Outlook connector
4. SendGrid / third-party API

**Decision**: Use Azure Logic App (Consumption) with the Microsoft Office 365 Outlook connector.

**Rationale**:
- No SMTP credentials required — connector uses OAuth2 with consented O365 account
- Logic App HTTP trigger provides a simple, secure integration surface
- Consumption plan means zero idle cost; pay only per trigger execution
- Demonstrates a common enterprise integration pattern (Logic Apps as middleware)
- Office 365 is ubiquitous in enterprise environments

**Consequences**:
- Requires a one-time consent from the O365 mailbox owner before deployment completes
- Logic App SAS URL must be stored in App Service app settings
- Email delivery latency adds ~2-5 seconds to the OTP flow (acceptable for demo)

## ADR-002: App Service → Logic App Authentication via SAS URL

**Status**: Accepted

**Context**: The App Service needs to securely call the Logic App HTTP trigger. Options:
1. Entra ID (AAD) token on the Logic App trigger
2. Logic App SAS URL (built-in trigger authentication)
3. API Management gateway
4. Unauthenticated trigger (unacceptable)

**Decision**: Use the Logic App's built-in SAS URL as the trigger endpoint, stored in the App Service app setting `LOGIC_APP_URL`.

**Rationale**:
- SAS URL includes a shared-access signature generated by Azure — no AAD app registration needed
- Simplest viable pattern for Consumption Logic Apps without Enterprise tier
- URL is non-guessable and can be regenerated if compromised
- Consistent with Microsoft's own documentation for Logic App HTTP trigger patterns

**Consequences**:
- SAS URL must be treated as sensitive and not committed to source control
- If the SAS URL is rotated, the App Service app setting must be updated
- Demo guide must document the SAS URL rotation procedure

> [!NOTE]
> Alternative: Configure Logic App trigger with AAD authentication and pass an Entra token from App Service using Managed Identity. This is more secure but adds complexity. Documented as an enhancement in the demo guide.

## ADR-003: Storage Access via System-Assigned Managed Identity

**Status**: Accepted

**Context**: The App Service needs write access to Azure Blob Storage. Options:
1. Storage account key (connection string)
2. SAS token generated per upload
3. System-Assigned Managed Identity with RBAC

**Decision**: System-Assigned Managed Identity with `Storage Blob Data Contributor` role on the storage account.

**Rationale**:
- Eliminates credential management entirely — no rotation, no secrets leakage
- Industry best practice (Microsoft Well-Architected Framework)
- `DefaultAzureCredential` in the SDK handles token acquisition transparently
- Demonstrates the zero-secret pattern effectively for demo audiences

**Consequences**:
- Storage account must have `allowSharedKeyAccess: false`
- Role assignment must be created in Bicep (Deployer also needs a role assignment)
- `DefaultAzureCredential` requires the Managed Identity to be system-assigned (not user-assigned for simplicity)

## ADR-004: Storage Private Endpoint with Disabled Public Access

**Status**: Accepted

**Context**: Storage security requirements. Options:
1. Public storage account with firewall rules
2. Service Endpoint (VNet-level restriction)
3. Private Endpoint with DNS Zone (fully private)

**Decision**: Private Endpoint for the blob subresource, with `publicNetworkAccess: Disabled`.

**Rationale**:
- Private Endpoint is the gold standard for storage isolation in enterprise scenarios
- Demonstrates a realistic enterprise deployment pattern
- App Service VNet Integration routes blob traffic privately through the VNet
- Private DNS Zone ensures the storage FQDN resolves to the private IP within the VNet

**Consequences**:
- Two subnets required: App Service delegation subnet + private endpoint subnet
- Private DNS Zone (`privatelink.blob.core.windows.net`) must be linked to the VNet
- Deployer cannot access blob data from their local machine after `publicNetworkAccess: Disabled` (documented workaround: temporarily enable public access during demo if direct portal exploration is needed)

## ADR-005: OTP Stored in Server-Side In-Memory Session

**Status**: Accepted

**Context**: OTP persistence options:
1. Azure Cache for Redis (distributed cache)
2. Azure Cosmos DB (document storage)
3. Azure Table Storage
4. ASP.NET Core in-memory session (single-instance)

**Decision**: ASP.NET Core in-memory session (single App Service instance).

**Rationale**:
- Zero infrastructure dependencies — no additional Azure service to provision
- Sufficient for demo: single App Service instance, short-lived OTPs (10 min)
- Simplifies the demo narrative (no cache/database in the OTP flow)
- OTPs are ephemeral by design — session expiry is the security mechanism

**Consequences**:
- Does not work correctly with multiple App Service instances (scale-out)
- If demo requires scale-out: add Azure Cache for Redis and switch to distributed session
- App Service must be configured with Always On to avoid session loss during cold starts

## ADR-006: B2 App Service Plan (Linux)

**Status**: Accepted

**Context**: App Service Plan SKU selection for demo.

**Decision**: B2 Linux App Service Plan.

**Rationale**:
- B2 supports VNet Integration (required for private storage access)
- Minimum SKU with VNet Integration capability on Linux
- Low cost for demo environments; easy to scale up for production guidance

**Consequences**:
- Free/F1 tier does NOT support VNet Integration — B2 is the minimum
- Always On is available at B2 (required for warm session handling)
