# Requirements — Secure OTP-Gated File Upload Portal

> Generated by requirements agent | 2026-04-06

## Executive Summary

A secure, enterprise-ready file upload portal where access is gated by a one-time password (OTP) delivered via email. The solution demonstrates Azure App Service, Azure Logic Apps (Office 365 connector), and Azure Blob Storage with private networking and managed identity — all without storing secrets or credentials in application code.

---

## Business Requirements

| ID   | Requirement                                                                                                           | Priority |
| ---- | --------------------------------------------------------------------------------------------------------------------- | -------- |
| BR-1 | Users must authenticate via a 6-digit OTP sent to their verified email before accessing the file upload portal        | Must     |
| BR-2 | Uploaded files must be stored durably and securely in cloud object storage                                            | Must     |
| BR-3 | No secrets, passwords, or storage keys may appear in application settings or source code                              | Must     |
| BR-4 | The solution must be deployable end-to-end with a single `azd up` command                                             | Must     |
| BR-5 | Email delivery must be handled by an enterprise-grade connector (Office 365 Outlook) via Azure Logic Apps             | Must     |
| BR-6 | The portal must demonstrate private networking: storage is not accessible from the public internet                    | Must     |

## Functional Requirements

### OTP Flow

| ID    | Requirement                                                                                              |
| ----- | -------------------------------------------------------------------------------------------------------- |
| FR-1  | User visits the portal and enters an email address on the landing page                                   |
| FR-2  | System generates a cryptographically random 6-digit numeric OTP and stores it in server-side session     |
| FR-3  | System calls Azure Logic App HTTP trigger with the email address and OTP as payload                      |
| FR-4  | Logic App sends an email via the Office 365 Outlook "Send an email" action to the provided address       |
| FR-5  | User enters the OTP on the verification page                                                             |
| FR-6  | System validates the OTP against the stored value; OTP expires after 10 minutes                         |
| FR-7  | On successful OTP validation, session is marked as authenticated and user is redirected to the upload page|

### File Upload

| ID    | Requirement                                                                                              |
| ----- | -------------------------------------------------------------------------------------------------------- |
| FR-8  | Authenticated users can upload one or more files via a form on the upload page                           |
| FR-9  | Uploaded files are streamed to Azure Blob Storage using the Azure SDK (no temporary disk writes)         |
| FR-10 | Files are stored in a dedicated container in the storage account                                         |
| FR-11 | App Service accesses Blob Storage exclusively via System-Assigned Managed Identity (no storage keys)     |
| FR-12 | Upload confirmation displays the filename and blob URL (container-scoped, not publicly accessible)       |

### Security

| ID    | Requirement                                                                                              |
| ----- | -------------------------------------------------------------------------------------------------------- |
| FR-13 | The App Service must call the Logic App using its SAS URL passed as an app setting (`LOGIC_APP_URL`)     |
| FR-14 | `LOGIC_APP_URL` is the only non-sensitive config item; all auth is handled by Managed Identity or AAD    |
| FR-15 | OTPs are never stored in persistent storage — only in in-memory session                                  |
| FR-16 | OTP input is rate-limited: max 5 attempts before session reset                                           |

## Architecture Requirements

### Azure Services

| Service                   | Role                                                                           | Tier / SKU                |
| ------------------------- | ------------------------------------------------------------------------------ | ------------------------- |
| Azure App Service         | Host .NET 10 C# Razor Pages web application                                    | B2 (demo; upgradeable)    |
| Azure App Service Plan    | Compute plan for App Service                                                   | B2 Linux                  |
| Azure Logic App           | Consumption-plan workflow; triggers on HTTP; sends OTP email via O365          | Consumption               |
| Azure Storage Account     | Blob container for uploaded files; private endpoint; no public access          | Standard LRS              |
| Azure Virtual Network     | Isolate resources; host private endpoints; App Service VNet integration        | /16 address space         |
| Subnet — App Service      | Delegated subnet for App Service VNet integration                               | /24                       |
| Subnet — Private Endpoints| Subnet for blob private endpoint                                               | /24                       |
| Private Endpoint (Blob)   | Private connectivity from App Service to Storage Account                       | blob subresource          |
| Private DNS Zone          | `privatelink.blob.core.windows.net` — resolve blob endpoint privately          | Linked to VNet            |
| Log Analytics Workspace   | Central log sink for all diagnostic settings                                   | PerGB2018                 |

### Mandatory Constraints

> [!IMPORTANT]
> These constraints are non-negotiable for the demo to reflect real enterprise security patterns.

1. **Storage Account**: `publicNetworkAccess: Disabled`, `allowBlobPublicAccess: false`, `allowSharedKeyAccess: false`
2. **Storage Access**: App Service uses System-Assigned Managed Identity with `Storage Blob Data Contributor` role
3. **Logic App trigger**: Secured via SAS URL; URL stored in App Service app settings as `LOGIC_APP_URL`
4. **No secrets in code**: `DefaultAzureCredential` for storage; Logic App SAS URL is non-sensitive token
5. **VNet**: Two subnets minimum — one for App Service delegation, one for private endpoints
6. **DNS**: Private DNS Zone for `privatelink.blob.core.windows.net` linked to VNet

## Out of Scope

- Multi-tenant or multi-user session isolation beyond server-side session
- File scanning / antivirus integration
- Azure AD B2C or full identity federation
- CDN or global distribution
- Database-backed OTP persistence (in-memory session is sufficient for the demo)
